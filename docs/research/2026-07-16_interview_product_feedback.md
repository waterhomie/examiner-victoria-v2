## 3. 详细产品决定

### ER-001｜建立“好用”的客观标准

当前不能只使用：

> 我自己觉得产品挺好用。

下一阶段至少从四个方面验证：

1. **流程完成**：用户能否顺利完成 Practice、Mock 和 Part 1—3；
2. **反馈质量**：纠错是否准确，Natural Version 是否保留原意；
3. **系统稳定**：录音、转写、模型调用或 TTS 失败后能否恢复；
4. **用户价值**：用户是否认为反馈有用，是否愿意继续下一题。

建议最小指标：

- 首次任务完成率；
- 平均响应时间；
- 反馈准确性；
- Natural Version 原意保留；
- 评分一致性；
- 异常恢复率；
- 用户反馈有用性评分；
- 下一题继续率。

### ER-002｜建立分层测试集

同一道题至少准备：

- 初学者回答；
- 中等水平回答；
- 高水平回答；
- 偏题回答；
- 回答过短；
- 无关输入；
- 语音识别错误文本。

重点检查：

- 是否对不同水平给出有区分的反馈；
- 是否对较好回答进行过度纠错；
- 是否对较差回答只给空泛建议；
- 是否对偏题和无效回答做合理处理；
- 评分是否随能力水平产生合理变化。

### ER-003｜验证垂直产品价值

现阶段可以成立的价值主张是：

> Examiner Victoria 将雅思口语考试流程、题库、追问、即时反馈、评分和复盘方式预设成可直接使用的产品流程，降低用户配置通用 AI 的门槛。

现阶段不能直接成立的说法是：

> Examiner Victoria 的效果比 ChatGPT Live 或成熟雅思产品更好。

建议统一竞品任务：

1. 完成一组 Part 1；
2. 完成一张 Part 2 题卡；
3. 完成一组 Part 3；
4. 输入一次过短回答；
5. 输入一次偏题回答；
6. 查看评分、纠错和表达优化；
7. 查看是否能导出和复盘。

对比维度：

- 首次使用配置成本；
- 是否稳定遵循雅思流程；
- 反馈结构；
- 评分可信度；
- 异常处理；
- 记录与复盘；
- 响应速度；
- 价格与使用限制。

### ER-004｜补充市场与竞品假设

需要回答：

- 大厂是否真的没有做类似产品？
- 若已经做了，用户为什么仍需要 Examiner Victoria？
- 若没有重点做，是需求不足、商业价值不足，还是实现成本过高？
- 目标用户是临时备考者还是长期学习者？
- 用户愿意为什么能力持续使用？
- 免费通用 AI 已经能做到多少？
- 微信小程序是否解决真实获客问题，还是只更换载体？

在这些问题没有验证前，微信小程序和商业化继续作为假设，不进入当前 P0 开发。

### ER-005｜建立题库治理流程

建议每条或每批题目增加：

```text
source_type: recall
source_channel: anonymized
source_date: YYYY-MM
import_date: YYYY-MM-DD
official: false
verification_status: unchecked / reviewed / corrected
version: v1
```

最小治理流程：

1. 导入前格式检查；
2. 重复题检测；
3. 明显错误和缺失字段检查；
4. 随机人工抽查；
5. 用户报告错误入口；
6. 修改后保留版本记录；
7. 对外明确“非官方回忆材料”。

### ER-006｜强化 Practice / Mock 模式定位

两种模式共享：

- 雅思 Part 1—3；
- 题库；
- 考试阶段状态机；
- 录音与转写；
- 考官问题；
- 最终评分和记录。

两种模式的核心差异不应只表述为“界面不同”：

| Practice | Mock |
|---|---|
| 目标是学习和纠正 | 目标是模拟考试状态 |
| 显示聊天和回答记录 | 考试中不显示完整聊天记录 |
| 每轮提供即时反馈 | 考试中不提供逐轮反馈 |
| 可以查看 Quick Correction 和 Natural Version | 结束后统一查看报告和 Transcript |
| 鼓励停下来理解和修改 | 鼓励连续完成考试流程 |

后续验证问题：

- 用户是否一眼理解两种模式的差异？
- Practice 的即时反馈是否会打断练习节奏？
- Mock 是否足够接近真实口试？
- 用户完成 Mock 后是否知道去哪里查看报告？

## 4. 不建议写入用户反馈表的内容

以下内容来自面试，但不应作为 Examiner Victoria 的用户需求记录：

### 1. “FastAPI 是 JS 服务、并发比 Python 好”

这不是准确的技术结论。FastAPI 本身是 Python Web 框架，不能作为项目反馈或架构依据。

### 2. “AI 时代 Idea 比 Code 重要”

这是职业与协作方法建议，不是产品需求。

可以写入个人学习或项目方法论，但不要写进用户反馈表。

### 3. “你是否看得懂 Codex 代码”

这是个人技术能力与协作流程问题。

可以形成内部工程行动项：

- 补状态机文档；
- 补前后端调用链；
- 建立 AI Coding Agent 验收清单。

但不属于用户反馈。

### 4. 教育内容推荐机制题

这是岗位能力考题，不是 Examiner Victoria 的直接反馈。

除非后续真的把产品扩展成自适应学习路径，否则不写入当前产品反馈日志。

### 5. 微信小程序与裂变获客

面试官是在复述你的规划，不代表用户已经提出该需求，也不构成市场验证。

## 6. 可以追加到 Core Findings 的三条结论

建议追加：

- Product quality needs an explicit evaluation set covering response speed, feedback accuracy, scoring consistency, user-level differentiation, and failure recovery.
- The product should not claim superiority over general AI or mature IELTS tools before same-task benchmarking and user validation.
- Recall-based question banks require source metadata, versioning, quality checks, and a visible non-official content boundary.
