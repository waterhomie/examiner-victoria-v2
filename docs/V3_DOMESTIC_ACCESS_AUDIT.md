# Examiner Victoria V3 国内可访问性审计与迁移规划

> **Historical migration audit.** This document preserves the pre-CloudBase risk assessment, including earlier Railway and gTTS assumptions. The migration has since advanced; use [V3 Current Status](V3_CURRENT_STATUS.md) for current deployment and provider facts.

> 本文档基于 V3 kickoff、V2 README/API/架构文档，以及当前 React + FastAPI 代码实现进行审计。本文只做访问链路、风险和迁移规划分析，不代表已经完成供应商、备案、价格或平台政策的官方核验。

## 1. 执行摘要

Examiner Victoria V3 当前最值得保护的资产是已经可用的 React H5 + FastAPI 单体产品闭环：用户打开网页后，可以完成题目加载、录音、音频上传、STT 转写、LLM 反馈、TTS 音频播放、报告生成和文本导出。V3 的国内公测目标不应从重构形态开始，而应先验证这个 H5 闭环在国内网络、微信内置浏览器、iOS Safari、Android Chrome/系统 WebView 中是否能稳定完成。

当前实现的主要风险不是前端代码复杂度，而是外部访问链路集中在境外或未核验服务上：Railway 承载前后端入口，LLM/STT 走 OpenAI-compatible provider，TTS 使用 gTTS，浏览器语音能力依赖客户端系统能力。只要其中任一关键环节在中国大陆网络、微信 WebView 或移动端权限策略下不稳定，就会造成公测链路中断。

推荐路线是路线 A 的保守版本：保留 React + FastAPI，先做国内 H5 最小闭环，逐个迁移关键依赖。第一阶段只迁移或验证访问入口，不同时替换前端、后端、LLM、STT、TTS。Railway 海外版应作为 V2.0.0/V3 beta 的可回滚基线保留，而不是作为国内公测唯一入口。

本轮不得直接根据记忆判断供应商价格、备案规则、平台最新政策。涉及 Railway、中国大陆云厂商、LLM/STT/TTS provider、微信小程序平台、ICP备案、公安备案、生成式 AI/深度合成/个人信息合规的内容，均标记为需要通过官方最新资料验证。

## 2. 当前架构

### 2.1 部署形态

当前项目是 React 前端与 FastAPI 后端的单仓库产品形态。生产部署倾向于由后端服务承载 API，并服务前端构建产物：

- 前端：React + Vite，构建为静态资源。
- 后端：FastAPI，提供 `/api` 路由和静态文件服务。
- 部署：以 Railway/Docker 为主要现有部署路径。
- 会话：前端维护练习状态，后端主要按请求处理，不依赖复杂账号系统或数据库。
- 外部 AI 能力：LLM、STT、TTS 分别由 provider adapter 或第三方服务完成。

这种架构对 V3 很有利：业务闭环集中、迁移面可控、可以先把入口和网络链路迁到国内可访问环境，再逐步替换 AI provider。

### 2.2 前端入口与构建

前端入口是 Vite/React 应用，主要职责包括：

- 渲染练习界面和流程状态。
- 调用后端 `/api`。
- 请求浏览器麦克风权限。
- 使用 MediaRecorder 或 WAV fallback 采集音频。
- 处理浏览器原生语音识别的可选路径。
- 播放后端返回的 TTS 音频。
- 在 localStorage 中保存可恢复的练习状态。
- 通过 Blob 下载文本报告、转写或练习记录。

目前未发现必须依赖 Google Fonts、jsDelivr、GitHub CDN 等运行时外部 CDN 的前端链路。静态资源主要应随前端构建产物自托管。

### 2.3 后端入口与路由

后端以 FastAPI 为入口，承担：

- API 路由注册。
- CORS 配置。
- 题目、练习、音频、报告等接口处理。
- LLM provider adapter 调用。
- STT 音频转写调用。
- TTS 音频生成调用。
- 静态前端文件服务。
- 基础限流、缓存或运行期状态。

当前未发现必须依赖 WebSocket 或全双工实时语音。主要是 HTTP 请求/响应模式，这降低了国内 H5 迁移难度。

### 2.4 LLM provider adapter

LLM 调用通过 OpenAI-compatible client/provider adapter 形式封装。示例和代码中出现的 base URL 指向 `https://api.gptsapi.net/v1`。这意味着实际可访问性取决于当前 provider 的网络、域名、服务稳定性、数据政策和计费规则。

该 provider 是否适合作为国内公测的正式依赖，需要通过官方最新资料验证，不能仅凭经验判断。

### 2.5 STT 调用链路

STT 服务端路径与 OpenAI-compatible provider 关系紧密，用户音频上传到后端后，由后端发送给 provider 完成转写。前端还存在浏览器原生 speech recognition 的可选路径，但不能作为唯一可用方案，尤其在 iOS、微信内置浏览器、部分 Android WebView 中兼容性不稳定。

因此，服务端 STT 是当前完整闭环的关键节点，也是国内访问风险重点。

### 2.6 TTS 调用链路

TTS 当前使用 gTTS。它是独立于 LLM/STT provider 的外部依赖。即使 LLM/STT 可访问，TTS 仍可能因为外部网络、服务策略、请求频率或音频格式问题成为阻断点。

TTS 对核心练习价值重要，但不是所有练习都必须由 TTS 才能完成。V3 第一阶段可以保留文本反馈和报告作为降级路径，并把 TTS 视为可降级能力。

### 2.7 Session 管理

当前 Session 更接近无账号、轻状态产品：

- 前端保存当前练习阶段、转写、反馈、报告等状态。
- localStorage 用于刷新恢复。
- 后端按请求接收上下文并返回结果。
- 暂无复杂账号、支付、长期数据库或多人协作状态。

这对国内公测很有利：不必在第一阶段引入账号体系、支付、复杂数据库或合规成本更高的数据闭环。

### 2.8 环境变量与配置

项目存在环境变量示例或配置名，覆盖：

- LLM/STT provider base URL、API key、model。
- CORS origin。
- 前端 API base URL。
- 部署端口。
- 运行环境。
- 可能的限流、缓存或遥测配置。

本审计未读取真实 `.env`，也不输出任何 Secret。

## 3. 完整访问链路

### 3.1 页面加载

当前实现：用户访问部署域名，Railway 上的 FastAPI/Docker 服务返回前端页面或静态入口。

服务提供方：Railway、项目后端服务、域名/DNS 服务商。

境外网络依赖：高。Railway 是否对中国大陆用户稳定，需要实测和官方资料核验。

中国大陆访问风险：高。首屏入口不可达会直接阻断全部产品能力。

微信内置浏览器风险：中到高。微信 WebView 对跳转、权限、证书、下载、媒体播放有额外行为差异。

iPhone/Android 风险：中。移动网络、DNS、证书链、TLS、缓存策略均可能影响首屏。

是否必须迁移：国内公测建议迁移或至少增加国内可访问入口。

是否可暂时保留：可以。Railway 应保留为海外基线和回滚环境。

可回滚方式：保留现有 Railway 域名、配置和分支；国内入口异常时切回 Railway 链路。

### 3.2 前端静态资源加载

当前实现：React/Vite 构建产物由同一后端或部署服务托管。

服务提供方：Railway/FastAPI 静态服务。

境外网络依赖：高，取决于入口服务位置。

中国大陆访问风险：高。JS/CSS/chunk 失败会导致白屏或功能缺失。

微信内置浏览器风险：中。缓存、MIME、HTTPS、跨域资源策略都需要验证。

iPhone/Android 风险：中。低端机和弱网下 chunk 加载失败需要降级提示。

是否必须迁移：如果选择国内 H5 公测，静态资源应随入口一起迁移到国内或国内可稳定访问环境。

是否可暂时保留：可以作为海外基线。

可回滚方式：保留当前构建产物和 Railway 静态服务。

### 3.3 后端 API 请求

当前实现：前端默认同源或通过配置调用 `/api`，FastAPI 返回 JSON 或音频数据。

服务提供方：FastAPI、Railway。

境外网络依赖：高。

中国大陆访问风险：高。API 不通将阻断题目、练习、STT、LLM、TTS、报告。

微信内置浏览器风险：中。HTTPS、CORS、请求超时、上传限制需要实测。

iPhone/Android 风险：中。弱网和后台切换会影响长请求。

是否必须迁移：建议第一阶段迁移或增加国内 API 入口。

是否可暂时保留：可以保留 Railway 作为回滚。

可回滚方式：前端 API base URL 或 DNS 回切到 Railway。

### 3.4 浏览器录音权限

当前实现：浏览器调用麦克风权限，使用 MediaRecorder；必要时使用 WAV fallback。

服务提供方：浏览器、操作系统、设备硬件。

境外网络依赖：无直接境外网络依赖。

中国大陆访问风险：低，但受浏览器和系统权限影响。

微信内置浏览器风险：高。微信 WebView 的录音权限、MediaRecorder 支持、用户手势要求和系统差异必须单独测试。

iPhone/Android 风险：高。iOS Safari 对录音、自动播放、后台切换更严格；Android 厂商 WebView 差异明显。

是否必须迁移：不一定迁移，但必须建立兼容性测试矩阵和降级说明。

是否可暂时保留：可以，前提是明确支持浏览器范围。

可回滚方式：提供文本输入或重新录制降级；保留服务端 STT 路径。

### 3.5 音频上传

当前实现：前端把录音文件上传到后端 API。

服务提供方：浏览器、FastAPI、部署网络。

境外网络依赖：高，取决于 API 入口。

中国大陆访问风险：高。跨境上传音频可能慢、失败或超时。

微信内置浏览器风险：中到高。上传大小、网络切换、页面后台化可能失败。

iPhone/Android 风险：中到高。音频 MIME、容器格式、文件大小差异需要覆盖。

是否必须迁移：若做国内公测，建议 API 和上传入口优先国内化。

是否可暂时保留：海外基线可保留。

可回滚方式：降低音频时长、压缩或切回海外入口。

### 3.6 STT 转写

当前实现：后端把音频发送给 STT provider 转写。

服务提供方：OpenAI-compatible provider 或其代理服务。

境外网络依赖：中到高，取决于 provider 实际部署、域名和网络。

中国大陆访问风险：高。provider 不稳定会导致核心练习中断。

微信内置浏览器风险：间接风险。浏览器只负责上传，实际风险在后端到 provider。

iPhone/Android 风险：间接风险。音频格式兼容会影响转写成功率。

是否必须迁移：不宜第一步立即迁移；应先验证入口和当前 provider，再决定替换。

是否可暂时保留：可以短期保留，但要有失败提示和成本保护。

可回滚方式：保留当前 provider adapter；新 provider 以配置开关接入。

### 3.7 LLM 生成反馈或问题

当前实现：FastAPI 调用 LLM provider 生成下一题、反馈、评分或报告内容。

服务提供方：OpenAI-compatible provider。

境外网络依赖：中到高，需官方和实测核验。

中国大陆访问风险：高。LLM 失败会直接影响核心体验。

微信内置浏览器风险：低到中，主要是长请求等待和用户切后台。

iPhone/Android 风险：中。长响应、弱网和页面生命周期会影响体验。

是否必须迁移：中期需要评估国内可访问 provider；第一阶段不建议与入口同时迁移。

是否可暂时保留：可以短期保留。

可回滚方式：provider adapter 配置化，保留原 provider。

### 3.8 TTS 语音生成

当前实现：后端使用 gTTS 生成语音音频。

服务提供方：gTTS 背后的外部服务。

境外网络依赖：高概率存在，需官方资料和实测核验。

中国大陆访问风险：高。TTS 可能成为独立阻断点。

微信内置浏览器风险：中。音频返回后播放仍受微信 WebView 限制。

iPhone/Android 风险：中到高。iOS 自动播放限制明显，需要用户手势触发。

是否必须迁移：建议作为第二阶段或第三阶段迁移，第一阶段先保留文本降级。

是否可暂时保留：可以，但必须可降级为文本题目/反馈。

可回滚方式：失败时跳过语音播放，显示文本。

### 3.9 音频播放

当前实现：前端播放后端返回的音频 URL 或 blob。

服务提供方：浏览器媒体能力、后端音频响应。

境外网络依赖：取决于音频是否来自同源后端。

中国大陆访问风险：中。若后端入口慢，音频加载慢。

微信内置浏览器风险：高。自动播放限制、静音策略、用户手势要求需要实测。

iPhone/Android 风险：高。iOS Safari 对非用户触发播放非常严格。

是否必须迁移：无需单独迁移，但要调整交互和降级。

是否可暂时保留：可以。

可回滚方式：提供点击播放按钮和文本显示。

### 3.10 报告与练习记录

当前实现：前端保留 session 数据，后端可生成报告，报告内容可展示和下载。

服务提供方：前端 localStorage、FastAPI、LLM provider。

境外网络依赖：报告生成依赖 LLM provider；本地记录不依赖境外网络。

中国大陆访问风险：中到高。LLM 不通时报告生成失败，但已完成转写和反馈的数据仍可展示。

微信内置浏览器风险：中。localStorage 稳定性、页面刷新、清理策略需验证。

iPhone/Android 风险：中。浏览器存储和内存限制需覆盖。

是否必须迁移：暂不需要迁移为数据库。

是否可暂时保留：建议保留轻状态，避免第一阶段引入账号和数据库。

可回滚方式：保留本地导出和 session 恢复。

### 3.11 导出或下载

当前实现：前端通过 Blob 生成 TXT 下载，包括报告、转写或练习记录。

服务提供方：浏览器下载能力。

境外网络依赖：无直接境外依赖。

中国大陆访问风险：低。

微信内置浏览器风险：高。微信内置浏览器对文件下载行为限制较多，可能需要复制文本、分享或打开系统浏览器作为替代。

iPhone/Android 风险：中到高。iOS 文件保存路径和下载提示不稳定。

是否必须迁移：不需要优先迁移，但必须提供替代动作。

是否可暂时保留：可以。

可回滚方式：显示可复制文本，提示在系统浏览器打开。

## 4. 国内可访问性风险

### 4.1 Railway 风险

Railway 当前是前端、后端、API、上传、静态资源的共同入口，因此是单点阻断风险。它是否能稳定面向中国大陆用户访问，需要用官方区域、服务条款、可用性说明和真实网络测试核验。

风险等级：高。

建议：保留 Railway 作为海外基线，但国内公测不应只依赖 Railway。

### 4.2 LLM provider 风险

当前 LLM 走 OpenAI-compatible provider。风险包括：

- 中国大陆访问稳定性未核验。
- provider 是否需要代理未核验。
- 数据处理、日志保存、模型来源、SLA 未核验。
- 价格、速率限制、滥用处理未核验。

风险等级：高。

建议：第一阶段先测通当前链路；第二阶段以 adapter 配置化方式逐个替换或增加国内可访问 provider。

### 4.3 STT provider 风险

STT 涉及用户音频上传和外部转写。风险包括：

- 音频跨境传输或外部处理的合规风险。
- 大文件上传和长请求超时。
- 音频 MIME 格式兼容。
- provider 成本和滥用风险。

风险等级：高。

建议：STT 是关键迁移候选，但不应与入口迁移同批进行。

### 4.4 TTS 风险

gTTS 是独立外部依赖，可能在国内网络、频率限制或服务策略上不稳定。

风险等级：中到高。

建议：短期保留但必须可降级；中期评估国内 TTS 或浏览器端方案。

### 4.5 外部 CDN 和硬编码 URL

当前审计未发现运行时必须加载 Google Fonts、jsDelivr、GitHub CDN 的前端依赖。代码中仍应持续检查硬编码外部 URL，尤其是 provider base URL、文档链接、示例 URL、音频或资源 URL。

风险等级：中。

建议：建立外部 URL 清单，并区分运行时依赖、开发依赖、文档链接。

### 4.6 CORS 与 HTTPS

当前 CORS 存在配置化空间。公共 beta 中如果允许 `*`，虽然便于测试，但会增加滥用和跨站调用风险。HTTPS 对录音权限、Service Worker、现代浏览器安全策略都是硬要求。

风险等级：中。

建议：国内入口确定后收紧 CORS allow origin；所有正式入口必须 HTTPS。

### 4.7 WebSocket 或流式响应

当前未发现必须依赖 WebSocket、SSE 或全双工语音的核心链路。这个点是迁移优势。

风险等级：低。

建议：V3 公测前不要引入全双工实时语音，避免扩大兼容性和成本风险。

### 4.8 音频格式兼容

MediaRecorder 在不同浏览器上支持的 MIME 差异较大。项目已有多格式尝试和 WAV fallback 思路，但仍必须实测：

- iOS Safari。
- Android Chrome。
- 微信 iOS 内置浏览器。
- 微信 Android 内置浏览器。
- 桌面 Chrome/Edge/Safari。

风险等级：高。

建议：第一阶段测试矩阵必须覆盖真实设备。

### 4.9 微信内置浏览器限制

微信内置浏览器风险集中在：

- 麦克风权限。
- MediaRecorder 支持。
- 上传稳定性。
- 自动播放限制。
- 文件下载限制。
- 长页面生命周期和后台行为。

风险等级：高。

建议：H5 公测应明确推荐系统浏览器；微信内先保证打开、说明、核心录音路径，再考虑深度适配。

### 4.10 iOS Safari 限制

iOS 风险集中在：

- 录音必须由用户手势触发。
- 音频播放必须由用户手势触发。
- 页面后台后录音或上传可能中断。
- Blob 下载和文件保存体验不稳定。

风险等级：高。

建议：设计上避免自动播放假设，提供手动播放和复制文本降级。

### 4.11 域名、备案和主体要求

如果使用中国大陆服务器和正式域名，通常会涉及 ICP 备案、公安备案、主体资质、云厂商接入规则等。具体要求随地区、主体、云厂商、服务类型变化，必须通过官方最新资料核对。

风险等级：高。

建议：V3 第一阶段可以并行准备合规核对清单，但不要因备案不确定而先重构产品形态。

### 4.12 隐私风险

项目处理的数据包括：

- 用户语音音频。
- 语音转写文本。
- IELTS 练习回答。
- LLM 反馈、评分和报告。
- 浏览器 localStorage 中的 session 内容。

风险等级：高。

建议：公测前至少补齐隐私说明、数据处理边界、第三方 provider 说明、删除/不保存策略和滥用防护。

### 4.13 API 成本和滥用风险

无账号、无付费、公开入口的 H5 产品容易被刷接口。音频上传、STT、LLM、TTS 都可能产生实质成本。

风险等级：高。

建议：第一阶段至少保留后端限流、请求大小限制、音频时长限制、错误预算和 provider key 保护。

## 5. 阻断点与非阻断点

### 5.1 最可能阻断点

- Railway 页面/API 入口：中国大陆访问不稳定会阻断全部链路。
- STT provider：音频上传后转写失败会阻断练习主流程。
- LLM provider：反馈、报告和问题生成失败会阻断核心价值。
- gTTS：语音生成失败会影响口语陪练体验，但可降级。
- 微信内置浏览器录音：权限或 MediaRecorder 不支持会阻断微信内完成练习。
- iOS 自动播放/录音策略：会影响语音交互体验。
- CORS/HTTPS 配置：正式域名切换时容易出现跨域或权限问题。

### 5.2 非立即阻断点

- 暂无账号系统：对小规模公测不是阻断点。
- 暂无支付：对作品集和公测不是阻断点。
- 暂无复杂数据库：短期不是阻断点，反而降低迁移复杂度。
- 暂无 WebSocket/全双工语音：不是缺陷，是国内 H5 迁移优势。
- TXT 导出在微信内不稳定：不是核心闭环阻断点，可用复制文本降级。

## 6. 路线 A / B / C 对比

| 维度 | 路线 A：保留 React + FastAPI，仅迁移到国内服务器 | 路线 B：React H5 + 国内云托管或 Serverless | 路线 C：微信小程序或小程序容器重构 |
| --- | --- | --- | --- |
| 与 V2 代码复用率 | 高，前后端和 API 基本复用 | 高到中，前端复用高，后端需适配云函数/托管 | 低到中，前端交互、录音、下载都需重写或适配 |
| 国内可访问性 | 高，取决于服务器、域名、备案和 provider | 高，取决于云平台、函数冷启动、网关限制 | 高，但受微信平台审核、类目、域名和接口限制 |
| 微信分享便利性 | 中，可分享 H5 链接 | 中到高，可配合 H5 分享卡片 | 高，天然在微信生态内 |
| 录音兼容性 | 中，需要真实设备测试 | 中，需要真实设备测试和云网关上传测试 | 中到高，小程序录音 API 更统一，但迁移成本大 |
| 部署难度 | 中，接近现有 Docker/FastAPI 模式 | 中到高，要适配 Serverless 限制 | 高，需要小程序工程、审核、域名白名单和接口适配 |
| 备案要求 | 大陆服务器通常需要核验 | 大陆云托管通常需要核验 | 小程序上线通常也涉及主体、域名和备案核验 |
| 成本 | 中，可预测但需云资源 | 低到中，初期可能便宜但受调用量影响 | 中，开发和维护成本较高 |
| 维护难度 | 低到中，最贴近现状 | 中，平台绑定更强 | 高，两套形态或重构维护成本高 |
| Codex 协作难度 | 低，现有代码结构清晰 | 中，需要理解云平台约束 | 高，需要小程序生态、构建和审核链路 |
| 回滚风险 | 低，Railway 可保留 | 中，平台差异可能导致回滚复杂 | 高，小程序重构与 H5 差异大 |
| 适合作品集程度 | 高，能展示完整 Web 产品和后端架构 | 高，能展示国内云部署能力 | 中到高，但容易把重点从产品能力转到平台适配 |
| 适合小规模公测程度 | 高，最快验证真实用户链路 | 高，但需要处理平台限制 | 中，适合后续微信内增长，不适合第一步 |

结论：路线 A 是当前最稳的主线，路线 B 可作为路线 A 的部署变体，路线 C 不适合作为第一阶段默认选择。

## 7. 推荐方案

### 7.1 推荐路线

推荐采用路线 A：保留 React + FastAPI，先完成国内 H5 最小闭环。

具体执行原则：

- 优先保留现有 React + FastAPI 产品能力。
- 优先完成国内 H5 可访问闭环。
- 先验证访问链路，再替换供应商。
- 不同时替换前端、后端、LLM、STT、TTS。
- 每次只迁移一个关键环节。
- V2.0.0/Railway 海外版始终可回滚。
- 不立即加入账号、支付、复杂数据库或全双工语音。

### 7.2 不推荐路线 B 作为独立主线的原因

路线 B 的 React H5 + 国内云托管或 Serverless 可以作为部署选项，但不建议在第一阶段把架构改造成深度 Serverless：

- 音频上传和长时间 STT/LLM 请求可能触发函数超时、网关大小限制或冷启动问题。
- FastAPI 当前已能承载完整 API，强行拆分会增加调试面。
- 国内可访问性首先取决于入口和 provider，不一定需要先改后端形态。

### 7.3 不推荐路线 C 立即启动的原因

微信小程序或小程序容器不是第一阶段推荐项：

- 代码复用率低，React H5 交互和小程序组件模型差异较大。
- 小程序录音 API 虽更平台化，但需要审核、类目、域名白名单、隐私协议等额外链路。
- 会同时引入平台迁移、录音迁移、发布审核、分享体系等多个变量。
- 当前最重要的是验证产品闭环和 AI provider 可用性，而不是先换容器。

## 8. 分阶段迁移计划

### 8.1 第一阶段：国内 H5 最小闭环

目标：在不替换核心业务代码和 AI provider 的前提下，验证国内用户能打开、录音、上传、转写、获得反馈、播放或降级、生成报告。

建议范围：

- 保留 React + FastAPI。
- 保留当前 LLM/STT/TTS provider。
- 新增或迁移一个国内可访问 H5 入口。
- 收紧并验证 CORS/HTTPS。
- 建立真实设备测试矩阵。
- 增加微信/移动端失败提示和降级路径。
- 保留 Railway 作为海外回滚环境。

不做：账号、支付、复杂数据库、小程序、全双工语音、同时替换多个 provider。

### 8.2 第二阶段：provider 风险拆分

目标：在入口已验证的情况下，逐个降低 AI provider 风险。

建议顺序：

1. LLM provider 可配置化和候选 provider 验证。
2. STT provider 可配置化和音频格式稳定性验证。
3. TTS provider 替换或降级策略优化。
4. 增加成本监控、请求限流和错误预算。

每次只切换一个 provider，并保留旧 provider 配置作为回滚。

### 8.3 第三阶段：微信生态适配评估

目标：基于 H5 公测数据决定是否进入小程序。

进入条件：

- 微信内置浏览器打开量明显高。
- H5 在微信内录音或下载限制成为增长瓶颈。
- 已明确主体、类目、备案、隐私协议和审核路径。
- 产品闭环已被真实用户证明值得迁移。

### 8.4 暂缓事项

- 账号系统。
- 支付系统。
- 复杂数据库。
- 全双工实时语音。
- 微信小程序重构。
- 大规模题库改造。
- Prompt 大改。
- 多 provider 同时切换。

## 10. 成本测算框架

成本不在本文直接定价，必须通过官方最新资料核验。建议按以下框架估算：

- 入口成本：服务器、带宽、对象存储或 CDN、HTTPS 证书、域名。
- 后端成本：CPU、内存、并发、日志、监控。
- STT 成本：按分钟、按请求或按音频大小计费。
- LLM 成本：按输入/输出 token、模型档位、报告生成频次计费。
- TTS 成本：按字符、请求、音频时长或第三方限制计费。
- 滥用成本：匿名用户刷接口、重复上传、恶意长音频。
- 运维成本：备案、监控、报警、故障回滚、日志留存。

建议第一阶段建立一个简单公式：

`单次完整练习成本 = STT 单次成本 + LLM 多轮成本 + TTS 成本 + 服务器/带宽摊销`

再按日活和人均练习次数估算：

`日成本 = DAU × 人均完整练习次数 × 单次完整练习成本 + 固定基础设施成本`

## 11. 备案与合规待核对项

以下事项必须通过官方最新资料、云厂商文档、平台规则或专业意见核验：

- 中国大陆服务器和域名是否需要 ICP 备案。
- 是否需要公安联网备案。
- 个人主体或企业主体是否可承载该类教育/AI 练习产品。
- 域名、服务器、CDN、对象存储是否需要同一备案主体或接入要求。
- 微信小程序上线所需主体、类目、隐私协议、用户信息处理规则。
- 用户音频是否属于敏感个人信息或需额外明示同意。
- 音频、转写文本、反馈内容是否会被第三方 provider 保存或用于训练。
- 跨境传输或境外 provider 处理个人信息的合规要求。
- 生成式 AI、深度合成、语音合成相关规定是否适用。
- 面向未成年人或教育场景是否有额外限制。
- provider SLA、价格、调用限制、可用地区和服务条款。

## 12. 回滚方案

### 12.1 产品回滚

- 保留 V2.0.0 或当前 Railway 海外版作为稳定基线。
- 国内入口异常时，通过 DNS、环境变量或发布说明切回 Railway。
- 保留当前 React + FastAPI 主流程，不在第一阶段重构核心业务。

### 12.2 provider 回滚

- LLM/STT/TTS provider 变更必须配置化。
- 每次只迁移一个 provider。
- 新 provider 上线前保留旧 provider 的环境变量和 adapter。
- provider 异常时允许降级：无 TTS 时显示文本；报告失败时保留转写和反馈。

### 12.3 数据回滚

- 第一阶段不引入复杂数据库，降低数据迁移风险。
- 前端 localStorage 结构如需变更，必须保留兼容读取。
- 报告导出仍保留文本下载或复制路径。

### 12.4 部署回滚

- 国内部署失败时不删除 Railway 配置。
- 不切换 Git remote，不切回 HTTPS。
- 不修改 V2 canonical worktree。
- 通过分支和环境变量控制 V3 beta 变更。
